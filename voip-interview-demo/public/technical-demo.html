Scenario 1: One-Way Audio Resolution

Question:

"Tell me about a time you resolved a one-way audio or dropped call issue using SIP traces and PCAPs. What did you find, and how did you fix it?"

Technical Approach:
  • Tools: sngrep + Wireshark for SIP/RTP analysis
  • Detection: RTP packets reaching only one direction
  • Root Cause: NAT configuration issue in FreeSWITCH
  • Solution: External IP correction + firewall rules
Analysis Phase
$ sngrep -d any -I pcap_capture.pcap SIP Request: INVITE sip:user@domain.com From: +5511912345678 To: +5511987654321 Call-ID: call-12345@freeswitch $ wireshark -r pcap_capture.pcap RTP Stream Analysis: SSRC: 0x12345678 (Direction: Outbound Only) Packets: 50 sent, 0 received Loss: 100%
Solution Implementation
<configuration name="sip-settings"> <param name="ext-rtp-ip" value="200.150.10.20"/> <param name="ext-sip-ip" value="200.150.10.20"/> <param name="rtp-ip" value="$${local_ip_v4}"/> </configuration> iptables -A INPUT -p udp --dport 10000:20000 -j ACCEPT iptables -A OUTPUT -p udp --sport 10000:20000 -j ACCEPT
Answer & Results:

"In one situation, I used sngrep and Wireshark to check the SIP signaling and noticed that RTP was not reaching one side because of a NAT configuration issue. After adjusting the external IP settings in FreeSWITCH and correcting the firewall rules, the audio started working normally again."

Scenario 2: Custom FreeSWITCH Dialplan

Question:

"Describe how you customized FreeSWITCH, OpenSIPS, or Kamailio for a non-standard call flow. What modules or scripts were involved?"

Technical Implementation:
  • Platform: FreeSWITCH with XML dialplan
  • Integration: Node.js backend for dynamic routing
  • Logic: Conditional routing based on database rules
  • Flexibility: Real-time call flow modifications
XML Dialplan Integration
<extension name="custom_routing"> <condition field="${destination_number}" expression="^(.+)$"> <action application="set" data="api_result=${jsapi(custom_routing.js ${destination_number})}"/> <action application="eval" data="${api_result}"/> </condition> </extension>
Node.js Backend Logic
// custom_routing.js function customRouting(destination) { const dbResult = database.query(` SELECT route, priority FROM routing_rules WHERE pattern = ? AND active = 1 `, [destination]); if (dbResult.length > 0) { return `transfer ${destination} XML ${dbResult[0].route}`; } return `transfer ${destination} XML default`; }
Answer & Architecture:

"I modified the XML dialplan and implemented conditional routing integrated with a Node.js backend service. The call decisions were handled in JavaScript based on system rules and database data, allowing dynamic behavior connected to the web application."

Scenario 3: Real-time ESL Event Processing

Question:

"Have you built a system reacting to live call events (ESL, Webhooks, AMI)? How did you handle external API calls, especially under failure or timeout conditions?"

Technical Architecture:
  • Event Source: FreeSWITCH ESL (Event Socket Library)
  • Backend: Node.js with WebSocket real-time updates
  • Events: CHANNEL_CREATE, CHANNEL_ANSWER, CHANNEL_HANGUP
  • Reliability: Timeout handling + error logging
ESL Connection Setup
const ESL = require('modesl'); const connection = new ESL.Connection('localhost', 8021, 'ClueCon'); connection.subscribe(['CHANNEL_CREATE', 'CHANNEL_ANSWER', 'CHANNEL_HANGUP']); connection.on('esl::event', (event) => { const eventData = event.getHeader('Event-Name'); const uuid = event.getHeader('Unique-ID'); // Process event and update UI processCallEvent(eventData, uuid, event); });
Error Handling Strategy
async function callExternalAPI(url, data, timeout = 5000) { try { const controller = new AbortController(); const timeoutId = setTimeout(() => controller.abort(), timeout); const response = await fetch(url, { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify(data), signal: controller.signal }); clearTimeout(timeoutId); return await response.json(); } catch (error) { console.error('API call failed:', error.message); logger.error('External API timeout/failure', { url, error }); return null; // Don't affect live call flow } }
Answer & Implementation:

"I built a Node.js service that listens to FreeSWITCH ESL events like CHANNEL_CREATE, CHANNEL_ANSWER, and CHANNEL_HANGUP. The backend processes these events, updates the application state, and the web interface is updated in real time using WebSockets so operators can track call status in the browser. When calling external APIs, I configured timeouts and error handling, and if something failed it was logged to avoid affecting the live call flow."
